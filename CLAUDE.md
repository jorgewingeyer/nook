# Nook — Project Documentation

## Quick Start

### Setup

1. Copy environment files:
   ```bash
   cp .env.local.example .env.local
   cp .dev.vars.example .dev.vars
   ```

2. Generate a secure `SESSION_SECRET` (minimum 32 characters):
   ```bash
   openssl rand -base64 32
   ```
   Update both `.env.local` and `.dev.vars` with this value.

3. Install dependencies:
   ```bash
   pnpm install
   ```

4. Set up the database:
   ```bash
   pnpm db:migrate:local
   pnpm db:seed:local
   ```

5. Start development server:
   ```bash
   pnpm dev
   ```

The dev server runs at `https://nook.localhost:1355` (HTTPS is automatic via Portless).

**Test login:** `admin@nook.local` / `password`

## Architecture

### Tech Stack

- **Framework:** Next.js 16 (App Router)
- **Runtime:** Cloudflare Workers via OpenNext adapter
- **Database:** Cloudflare D1 (SQLite) + Drizzle ORM
- **Storage:** Cloudflare R2
- **UI:** React 19, Tailwind CSS 4, shadcn/ui, Radix UI
- **Auth:** JWT (HS256) with PBKDF2 password hashing
- **Language:** TypeScript with strict mode

### Key Design Decisions

1. **No API Routes** — All mutations use Next.js Server Actions (colocated or in domain-specific folders)
2. **Serverless-First** — Designed for Cloudflare Workers via OpenNext adapter; local dev uses Wrangler emulation
3. **JWT Auth** — Stateless sessions via `jose` + httpOnly cookies (no database sessions)
4. **D1/R2 Bindings** — Cloudflare resources injected via Wrangler; accessed through `src/lib/db.ts` and `src/lib/storage.ts`
5. **React Compiler** — Enabled automatically for performance optimization (see `babel-plugin-react-compiler` in package.json)

### Folder Structure

```
src/
  app/                         # Next.js App Router
    (auth)/                   # Public auth pages (login, change-password)
      login/
        page.tsx              # Login page
        login.action.ts       # Login Server Action
      change-password/        # Similar structure
      layout.tsx
    (secure)/                 # Protected pages (require authentication)
      page.tsx                # Dashboard
      layout.tsx
    globals.css
    layout.tsx                # Root layout
  components/                 # React components (to be created)
    ui/                       # shadcn/ui components (auto-generated)
  db/
    schema.ts                 # Drizzle schema definitions
    seed.sql                  # Local test data
    seed.prod.sql             # Production seed (empty)
  lib/
    constants.ts              # Environment constants
    db.ts                     # Drizzle ORM instance
    session.ts                # JWT encryption/decryption, password hashing
    storage.ts                # R2 operations wrapper
    utils.ts                  # Utilities (cn() for tailwind merging)
  middleware.ts               # Route protection and redirects
  scripts/
    db-fresh.ts               # Database reset script
drizzle/                      # Database migrations (auto-generated by Drizzle)
```

## Database Schema

Currently one table:

### users
- `id` — Primary key
- `name` — User's full name
- `email` — Unique email
- `password` — PBKDF2-SHA256 hash
- `role` — Enum: `admin` | `agent`
- `isActive` — Boolean flag
- `mustChangePassword` — Forces password change on next login
- `createdAt` — Unix timestamp

**Migrations:** Stored in `drizzle/` directory. Generated via `pnpm db:generate`, applied with `pnpm db:migrate:local`.

---

## Authentication Flow

1. User logs in at `/login` → `loginAction()` verifies email/password
2. Session cookie set with JWT payload: `{ userId, role, mustChangePassword, expiresAt }`
3. Middleware checks every request; protects `(secure)/*` routes
4. If `mustChangePassword` is true, user redirected to `/change-password`
5. Admin routes (`/users`, `/settings`) check `role === "admin"`
6. Sign out deletes session cookie

**Password Hashing:** PBKDF2-SHA256 (100k iterations), salted with `SESSION_SECRET`

## Environment Variables

### `.env.local` and `.dev.vars` (Development)

- **SESSION_SECRET** (min. 32 chars) — Used for JWT signing and password salt. Must match in both files.

### `wrangler.jsonc` (Cloudflare Configuration)

**Bindings** (automatically injected into the Worker):
- **DB** — D1 database binding (local emulation via `.wrangler/state/v3/d1/`)
- **R2_BUCKET** — R2 bucket binding (local emulation via `.wrangler/state/v3/r2/`)
- **ASSETS**, **IMAGES**, **WORKER_SELF_REFERENCE** — OpenNext adapter bindings (do not modify)

**Variables**:
- **NEXT_PUBLIC_CDN_URL** — Public URL for R2 assets (e.g., `https://assets.nook.local` in dev)

### Updating Cloudflare Resources for Production

After creating D1 and R2 in Cloudflare dashboard:

1. Update `database_id` in `wrangler.jsonc` under `d1_databases[0].database_id`
2. Update `bucket_name` in `wrangler.jsonc` under `r2_buckets[0].bucket_name`
3. Update `NEXT_PUBLIC_CDN_URL` if using a custom R2 domain
4. Regenerate types: `pnpm cf-typegen` (generates `cloudflare-env.d.ts`)

## Code Conventions

- **Server Actions:** Named `*.action.ts`, placed colocated with pages or organized by domain. Return `{ error?: string }` or `{ error?: string, data?: T }`.
- **Components:** Use shadcn/ui from `@/components/ui`. Create custom components in `src/components/` as needed.
- **Styling:** Tailwind CSS 4 + `cn()` utility from `@/lib/utils` for merging classes.
- **Database:** Always use `getDb(env.DB)` to access Drizzle instance; schema in `@/db/schema.ts`.
- **Type Safety:** Strict TypeScript mode enforced; use type inference where possible.
- **No Comments:** Only document non-obvious logic or workarounds.
- **Path Alias:** `@/` resolves to `src/`.

## Creating New Features

### Adding a New Page

1. Create folder under `src/app/(secure)/feature-name/` or `src/app/(auth)/` depending on access level
2. Add `page.tsx` as the page component
3. If needed, colocate a `feature-name.action.ts` Server Action in the same folder
4. Protect routes automatically via middleware (checks for valid session)

### Creating a Server Action

```typescript
// src/app/(secure)/feature/feature.action.ts
"use server";

import { COOKIE_SESSION } from "@/lib/constants";
import { cookies } from "next/headers";

export async function featureAction(formData: FormData) {
  const session = cookies().get(COOKIE_SESSION)?.value;
  if (!session) {
    return { error: "Unauthorized" };
  }

  try {
    // Your logic here
    return { data: result };
  } catch (error) {
    return { error: error instanceof Error ? error.message : "Unknown error" };
  }
}
```

### Adding a Database Table

1. Update `src/db/schema.ts` with Drizzle table definition
2. Run `pnpm db:generate` to create migration file
3. Run `pnpm db:migrate:local` to apply to local database
4. Use `getDb(env.DB)` in Server Actions to query

## Development Runtime Details

### Portless (HTTPS Local Dev)

`pnpm dev` uses **Portless**, which provides HTTPS proxying for local development. This is required because:
- Cloudflare Workers runtime requires HTTPS
- Modern browser APIs (cookies, service workers) enforce HTTPS
- Portless binds to `nook.localhost:1355` and handles TLS automatically

No additional setup needed; it runs as part of the dev script.

### OpenNext Adapter

The project uses **OpenNext** (`@opennextjs/cloudflare`) to compile Next.js for Cloudflare Workers:
- `pnpm build` outputs `.open-next/` directory (intermediate build artifacts)
- `pnpm preview` starts a local Cloudflare Workers runtime with the built app
- `pnpm deploy` builds and deploys to Cloudflare

Local dev (`pnpm dev`) does NOT use the OpenNext build; it runs Next.js directly with Wrangler emulation.

### Local Database Emulation

Wrangler emulates D1 (SQLite) and R2 (object storage) locally:
- **D1:** `.wrangler/state/v3/d1/` stores SQLite database file
- **R2:** `.wrangler/state/v3/r2/` stores object files

These directories are git-ignored; each `pnpm dev` session starts fresh from migrations.

### Drizzle Studio

Run `pnpm db:studio` to open an interactive UI for viewing and editing your local D1 database. Useful for inspecting schema, running raw SQL, or manually editing data during development.

## Common Tasks

### Add a shadcn/ui Component
```bash
pnpm dlx shadcn@latest add button
```
This generates components in `src/components/ui/` and updates `tailwind.config.js`.

### View/Edit Database
```bash
pnpm db:studio
```
Opens Drizzle Studio in the browser for interactive database exploration.

### Reset Database to Fresh State
```bash
pnpm db:migrate:fresh
```
Drops all tables and reapplies migrations; useful when schema changes break existing data.

### Check TypeScript Errors
TypeScript is validated on every build and lint, but to check manually:
```bash
npx tsc --noEmit
```
